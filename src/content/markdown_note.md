Markdown 是一种轻量级标记语言。它用少量符号表达文章结构，让写作者能够专注于内容，而不必在排版工具栏之间频繁切换。

这篇教程将从最常用的基础语法开始，逐步介绍 GitHub Flavored Markdown（GFM）扩展和一些实用的变通技巧。读完后，你可以独立编写技术文档、项目说明和博客文章。

## 为什么选择 Markdown

Markdown 文件本质上是纯文本，因此具有几个明显优势：

- **容易阅读**：即使没有渲染，源文件仍然清晰可读。
- **便于管理**：可以直接使用 Git 记录每一次修改。
- **跨平台**：编辑器、代码托管平台和静态博客通常都支持 Markdown。
- **适合技术内容**：代码块、链接、图片和表格都能自然地融入正文。

不过，不同平台使用的 Markdown 解析器并不完全相同。基础语法通常具有良好的兼容性，而 HTML、任务列表、警告框等扩展功能需要根据目标平台进行测试。

## 标题与段落

使用 `#` 创建标题。`#` 的数量表示标题层级：

```md
# 一级标题
## 二级标题
### 三级标题
#### 四级标题
```

一篇文章通常只需要一个一级标题。正文中的主要章节使用二级标题，章节内部的小节使用三级标题。

段落之间应保留一个空行：

```md
这是第一段。

这是第二段。
```

在当前博客中，文章标题会根据 frontmatter 自动显示，因此正文可以直接从介绍段落或二级标题开始。二级和三级标题还会被自动收集到文章目录中。

## 强调、删除线与分隔线

Markdown 使用成对符号表达强调：

```md
这是 **粗体文字**。
这是 *斜体文字*。
这是 ***粗斜体文字***。
这是 ~~已经删除的内容~~。
```

渲染效果：

这是 **粗体文字**，这是 *斜体文字*，这是 ~~已经删除的内容~~。

使用三个或更多连字符可以创建分隔线：

```md
---
```

不要过度使用强调。一个段落中同时出现大量粗体、斜体和颜色，反而会削弱视觉重点。

## 列表与任务清单

无序列表可以使用 `-`、`*` 或 `+`，建议在同一份文档中统一使用一种写法：

```md
- HTML
- CSS
- JavaScript
```

有序列表使用数字和句点：

```md
1. 明确文章主题
2. 编写内容提纲
3. 完成正文
4. 校对并发布
```

列表可以嵌套，但子项目需要缩进：

```md
- 前端开发
  - React
  - Vue
- 后端开发
  - Node.js
  - Go
```

GFM 还支持任务清单：

```md
- [x] 完成初稿
- [x] 检查代码示例
- [ ] 发布文章
```

- [x] 完成初稿
- [x] 检查代码示例
- [ ] 发布文章

## 引用与提示信息

在段落前添加 `>` 可以创建引用：

```md
> 好的文档不仅记录结果，也应该解释关键决策。
```

> 好的文档不仅记录结果，也应该解释关键决策。

标准 Markdown 没有统一的警告框语法，但可以用引用块模拟提示信息：

```md
> **注意：** 不同 Markdown 平台支持的扩展语法可能不同。

> **提示：** 发布前应在目标平台检查渲染结果。

> **警告：** 不要在公开文档中提交密码或 API 密钥。
```

这种写法只依赖基础语法，比平台专用的警告指令更容易迁移。

## 链接与图片

行内链接由链接文字和地址组成：

```md
[访问 Markdown 中文教程](https://markdown.com.cn/)
```

当一个地址需要多次使用时，可以改用引用式链接：

```md
阅读 [Markdown 教程][markdown-guide] 学习更多内容。

[markdown-guide]: https://markdown.com.cn/
```

图片语法与链接相似，只是在开头增加 `!`：

```md
![图片替代文字](/images/example.png)
```

替代文字应描述图片表达的信息，而不是简单写“图片”。它可以帮助屏幕阅读器用户，也能在图片加载失败时提供上下文。

在这个博客项目中，建议把图片存放到 `public/images/`，然后通过 `/images/文件名.png` 引用。

### 为图片添加标题

Markdown 没有统一的图片标题语法。兼容性较好的做法是在图片下一行添加斜体说明：

```md
![响应式布局示意图](/images/responsive-layout.png)

*图 1：桌面端与移动端的布局变化。*
```

支持原始 HTML 的平台也可以使用 `<figure>` 和 `<figcaption>`，但这种方式并不能保证在所有 Markdown 环境中生效。

## 行内代码与代码块

使用反引号标记行内代码：

```md
使用 `npm run dev` 启动开发服务器。
```

多行代码使用三个反引号包围，并在开头标明语言：

````md
```javascript
function greet(name) {
  return `Hello, ${name}!`;
}
```
````

语言名称会帮助渲染器进行语法高亮。常用标识包括 `javascript`、`typescript`、`jsx`、`css`、`html`、`json`、`bash` 和 `python`。

编写代码示例时应注意：

- 保留能够说明问题的最小代码。
- 明确代码运行所需的环境。
- 不要在示例中包含真实令牌、密码或私人地址。
- 对不明显的关键步骤补充简短说明。

## 表格

GFM 使用竖线和连字符创建表格：

```md
| 语法 | 用途 | 推荐场景 |
| --- | --- | --- |
| `**text**` | 粗体 | 强调关键词 |
| `` `code` `` | 行内代码 | 命令与变量名 |
| `[title](url)` | 链接 | 引用外部资料 |
```

| 语法 | 用途 | 推荐场景 |
| --- | --- | --- |
| `**text**` | 粗体 | 强调关键词 |
| `` `code` `` | 行内代码 | 命令与变量名 |
| `[title](url)` | 链接 | 引用外部资料 |

可以使用冒号控制对齐：

```md
| 左对齐 | 居中 | 右对齐 |
| :--- | :---: | ---: |
| A | B | C |
```

### 表格内换行与列表

基础 Markdown 不支持在单元格内直接写多段内容。一些允许 HTML 的平台可以使用 `<br>` 换行：

```md
| 项目 | 说明 |
| --- | --- |
| 发布 | 完成构建。<br>推送到远程仓库。 |
```

复杂列表也可以使用 HTML 的 `<ul>` 和 `<li>`，但当单元格内容越来越复杂时，普通标题加列表通常比表格更容易阅读和维护。

## 转义特殊字符

当你想显示 Markdown 符号本身时，可以在符号前添加反斜杠：

```md
\*这段文字不会变成斜体\*
\# 这也不会变成标题
```

常见的可转义符号包括：

```text
\  `  *  _  { }  [ ]  < >  ( )  #  +  -  .  !  |
```

版权符号、箭头等字符可以直接输入，也可以使用 HTML 实体：

```md
&copy; 2026
&rarr; 下一步
&#176;C
```

## 注释与隐藏说明

Markdown 没有标准注释语法。某些解析器会忽略引用定义，因此可以利用未使用的链接定义保存编辑备注：

```md
[这里是不会显示的编辑备注]: #
```

支持 HTML 的平台通常也支持 HTML 注释：

```html
<!-- 这是一条编辑备注，不会显示在页面中。 -->
```

需要注意，隐藏只代表不在渲染页面中显示，注释仍然存在于源文件里。不要把敏感信息写入 Markdown 注释。

## 下划线、居中与文字颜色

Markdown 有意减少视觉样式控制，因此没有标准的下划线、居中和文字颜色语法。如果平台允许原始 HTML，可以使用以下写法：

```html
<ins>带下划线的内容</ins>

<p style="text-align: center;">居中的段落</p>

<span style="color: #0ea5e9;">带颜色的文字</span>
```

这些写法的可移植性较差，而且部分平台会出于安全原因过滤 HTML 或内联样式。对于普通博客文章，优先使用标题、粗体、引用和结构层级表达重点。

当前博客使用 `react-markdown` 安全渲染内容，默认不会执行 Markdown 中的原始 HTML。因此，上面的 HTML 代码适合作为教程示例，但不会直接改变本站文章的样式。

## 调整图片尺寸

标准图片语法不能指定宽高。支持 HTML 时可以使用：

```html
<img src="/images/example.png" alt="示例图片" width="480">
```

更推荐的做法是在网站样式表中统一控制图片最大宽度，让图片自动适配手机和桌面屏幕。这样文章内容不需要保存大量与具体设备相关的尺寸数据。

## 创建文章目录

如果平台不支持自动目录，可以手动链接到标题锚点：

```md
## 目录

- [基础语法](#基础语法)
- [代码块](#代码块)
- [表格](#表格)
```

标题锚点的生成规则可能因平台而异，尤其是中文、空格和特殊符号的处理方式。手动目录发布前应逐项点击检查。

当前博客会自动提取二级和三级标题，生成桌面端侧边目录和移动端折叠目录，所以文章中不需要再手写目录。

## 嵌入视频

Markdown 本身不能直接播放视频。兼容性最好的方式是使用一张预览图链接到视频网站：

```md
[![视频预览图](/images/video-cover.jpg)](https://example.com/video)
```

允许 HTML 的平台可以使用 `<video>` 或视频网站提供的 `<iframe>`，但需要考虑以下问题：

- 平台是否会过滤相关标签。
- 视频在移动网络下的加载成本。
- 第三方播放器的隐私和 Cookie。
- 是否提供标题、字幕和文字说明。

对技术文章而言，链接到视频并提供文字摘要，通常比只嵌入播放器更可靠。

## 一份适合博客的文章模板

下面的模板可以直接用于这个博客：

```md
---
title: "文章标题"

tags: ["tag1", "tag2"]
category: "技术"
featured: "false"
draft: "false"
excerpt: "用一两句话概括文章解决的问题。"
---

这里写文章简介，说明背景和读者能获得什么。

## 问题背景

介绍问题和使用场景。

## 解决方案

解释核心思路，并提供必要的代码示例。

### 关键步骤

拆解实现过程。

## 常见问题

记录限制、边界情况和排查方式。

## 总结

提炼可以复用的结论。
```

## 写作与兼容性建议

为了让文章长期可维护，可以遵循以下原则：

1. **优先使用基础语法**：基础标题、列表、链接和代码块具有更好的可移植性。
2. **保持标题层级连续**：不要从二级标题直接跳到四级标题。
3. **提供有意义的链接文字**：避免大量使用“点击这里”。
4. **为图片填写替代文字**：描述图片传达的信息。
5. **控制表格复杂度**：移动端很难阅读过宽、过深的表格。
6. **谨慎使用 HTML**：先确认目标平台是否允许并正确渲染。
7. **发布前预览**：检查目录、链接、代码高亮和移动端布局。

Markdown 的价值不在于提供所有排版能力，而在于用稳定、清晰的文本结构保存知识。当基础语法不足时，可以使用少量 HTML 变通，但应该把兼容性和可维护性放在视觉效果之前。
